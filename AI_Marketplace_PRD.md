> **문서 버전**: 1.0  
> **작성일**: 2024년 12월

## 1. 목표

**AI 업무자동화 솔루션(n8n 템플릿, AI Agent, 바이브코딩 앱)을 거래할 수 있는 신뢰 기반 마켓플레이스 플랫폼 구축**

---

## 2. 배경 및 문제 정의

### 2.1 현재 문제
- AI 자동화 솔루션 판매자: 판매 채널 부재, 신뢰 구축 어려움
- AI 자동화 솔루션 구매자: 품질 검증 불가, 맞춤 솔루션 찾기 어려움
- 바이브코더: 비전문가도 만들 수 있지만 판매할 곳이 없음

### 2.2 기회
- AI 시대 새로운 크리에이터 이코노미 형성
- 전문가 검증 시스템으로 차별화 가능
- 다양한 수익 모델(일회성, 구독, SaaS) 지원

## 3. 타겟 유저

### 3.1 판매자 (Seller)
| 유저 타입 | 특징 | 니즈 |
|----------|------|------|
| n8n/Make 전문가 | 노코드 워크플로우 개발 | 템플릿 판매로 수익화 |
| AI Agent 개발자 | LangChain, CrewAI 등 활용 | 에이전트 솔루션 판매 |
| 바이브코더 | AI 활용 코딩, 비전문 개발자 | 쉬운 상품 등록, 검증 지원 |
| 프리랜서/소규모 개발사 | 맞춤 개발 가능 | 개발 의뢰 수주 |

### 3.2 구매자 (Buyer)
| 유저 타입 | 특징 | 니즈 |
|----------|------|------|
| 중소기업/스타트업 | 업무 자동화 필요 | 검증된 솔루션 빠른 도입 |
| 마케팅/영업/CS 담당자 | 반복 업무 많음 | 즉시 적용 가능한 템플릿 |
| 1인 기업가/프리랜서 | 시간 부족 | 저렴한 자동화 솔루션 |
| DX 추진 기업 | 디지털 전환 중 | 맞춤 개발 의뢰 |

## 4. 유저 스토리

### 4.1 판매자 스토리
```
US-S1: 판매자로서, n8n 워크플로우를 상품으로 등록하고 싶다.
       → 가격/설명/파일 업로드 후 즉시 판매 시작

US-S2: 판매자로서, 내 상품의 신뢰도를 높이기 위해 전문가 검증을 받고 싶다.
       → 검증 요청 → 검증자 리뷰 → Level 1-3 배지 획득

US-S3: 판매자로서, 판매 현황과 정산 내역을 확인하고 싶다.
       → 대시보드에서 매출/정산/리뷰 통합 조회

US-S4: 판매자로서, 개발 의뢰를 받아 맞춤 개발을 하고 싶다.
       → 의뢰 목록 확인 → 제안서 제출 → 선정 시 개발 진행
```

### 4.2 구매자 스토리
```
US-B1: 구매자로서, 내 업무에 맞는 자동화 솔루션을 검색하고 싶다.
       → 카테고리/태그/검증레벨 필터로 검색

US-B2: 구매자로서, 검증된 상품을 안심하고 구매하고 싶다.
       → 검증 레벨, 리뷰, 판매자 등급 확인 후 결제

US-B3: 구매자로서, 원하는 솔루션이 없으면 개발을 의뢰하고 싶다.
       → 요구사항 작성 → 개발자 제안 수신 → 선택 및 에스크로 결제

US-B4: 구매자로서, 구매한 상품에 문제가 있으면 환불받고 싶다.
       → 7일 이내 환불 요청 → 분쟁 조정 → 환불 처리
```

### 4.3 검증자 스토리
```
US-V1: 검증자로서, 검증 요청을 받아 코드 리뷰를 수행하고 싶다.
       → 검증 요청 목록 → 코드 분석 → 보고서 작성 → 수수료 수령
```

```
Users
├── id (UUID, PK)
├── email (UNIQUE)
├── role (buyer | seller | verifier | admin)
├── seller_tier (new | verified | pro | master)
└── created_at, updated_at

Products
├── id (UUID, PK)
├── seller_id (FK → Users)
├── title, description
├── category (n8n | make | ai_agent | app | api | prompt)

├── verification_level (0-3)
├── status (draft | pending | active | suspended)
└── created_at, updated_at

Orders
├── id (UUID, PK)
├── buyer_id (FK → Users)
├── product_id (FK → Products)
├── amount, currency
├── status (pending | paid | completed | refunded)
└── created_at

Reviews
├── id (UUID, PK)
├── order_id (FK → Orders)
├── rating (1-5)
├── comment
└── created_at

Verifications
├── id (UUID, PK)
├── product_id (FK → Products)
├── verifier_id (FK → Users)
├── level (1-3)
├── status (pending | in_progress | approved | rejected)
```

## 10. 수용 기준 (Acceptance Criteria)

### 10.1 상품 등록 (US-S1)
```gherkin
Given 인증된 판매자가 로그인했을 때
When 상품 등록 폼을 작성하고 제출하면
Then 상품이 "pending" 상태로 저장되고
And 관리자 승인 후 마켓에 노출된다
And 판매자 대시보드에서 상품을 확인할 수 있다
```

### 10.2 상품 구매 (US-B2)
```gherkin
Given 인증된 구매자가 상품 상세 페이지에 있을 때
When 결제 버튼을 클릭하고 결제를 완료하면
Then 주문이 생성되고 상태가 "paid"가 되고
And 구매자에게 다운로드 링크가 제공되고
And 판매자에게 알림이 발송된다
```

### 10.3 전문가 검증 (US-S2)
```gherkin
Given 판매자가 검증 요청을 제출했을 때
When 검증자가 코드 리뷰를 완료하면
Then 검증 보고서가 생성되고
And 상품에 검증 레벨 배지가 표시되고
And 검증자에게 수수료의 70%가 정산된다
```

```
  2) API 엔드포인트 구현

  4) 테스트 작성
  5) 리팩터링

출력형식: 파일 트리 + 변경 파일별 코드 + 실행 방법
```

## 부록 B: 바이브코딩 5원칙

1. **요구사항 먼저**: "뭘 만들지" 불명확하면 AI는 그럴듯한 쓰레기를 빠르게 만듦
2. **한 번에 1기능**: 한 번에 크게 만들면 엣지케이스가 무너짐
3. **즉시 검증**: "컴파일/실행/테스트 통과"가 유일한 진실
4. **컨텍스트 관리**: 파일/모듈/규칙/제약을 "AI가 항상 보는 곳"에 배치
5. **의도적 리팩터링**: 생성 → 동작 확인 → 정리 순서
